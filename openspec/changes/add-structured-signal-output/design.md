## 背景

项目目前已经具备三层比较稳定的基础设施：

- `efinance_cli/indicators/` 下的可复用技术指标计算层
- `efinance_cli/enrichment/` 下的市场数据增强链路
- `efinance_cli/rendering.py` 下的统一输出渲染层

当前这三层的连接方式，更适合“增强后的原始指标输出”，而不适合“结构化观察输出”。执行器会先调用上游 `efinance`，再增强结果，最后直接渲染返回对象。这种方式适合补指标列，但不能形成稳定的近期事件检测、近期 trace 检视和跨格式统一表达契约。

目前的 table 渲染主要依赖 `DataFrame` 的直接字符串化。一旦增强列增多，输出就会迅速变宽、难扫读，也不符合用户希望先看连续 32 根现象、再看事件总结、并以规整 ASCII 盒子方式呈现的交互目标。

本设计必须满足以下硬约束：

- 复用现有指标计算能力
- 默认输出只陈列事实，不给主观方向性判断
- 新的结构化观察输出路径中，CLI 输出语言必须统一为英文
- `table`、`json`、`csv`、`tsv` 必须具备等价信息量
- 除 `trace_points` 外，优先采用纵向布局而不是横向宽表
- `trace_points` 允许横向数值块，但必须仍然是浮点数而非字符图
- 默认 trace window 为 32，并允许用户覆盖
- table section 必须使用统一、规整、动态宽度、不可穿墙的 ASCII 矩形框

## 目标 / 非目标

**目标：**

- 为支持的最新行情和历史行情命令引入结构化观察数据模型。
- 在指标增强之后增加一层可复用的近期事件检测层，将连续指标序列转换成事实型事件记录。
- 引入独立的 trace window 机制，默认值为 32。
- 为结构化观察视图提供规整的 boxed table 渲染能力。
- 让 `json`、`csv`、`tsv` 与 `table` 保持等价信息量，即使序列化形态不同。
- 在保留现有 raw/enriched 路径的同时，新增 observation-oriented 输出路径。

**非目标：**

- 不把新结构化观察视图一次性替换为所有命令的默认输出。
- 不在默认输出里加入 bullish、bearish、confidence、regime 一类预判性汇总。
- 不改造不具备兼容历史增强能力的无关市场数据命令。
- 不引入图表、unicode sparkline 或图形化 trace。
- 不引入新的第三方终端 UI 依赖。

## 设计决策

### 决策一：引入显式的 observation payload 模型

实现应在 enrichment 与 rendering 之间增加一层内部结构化 payload，仅对支持的命令生效。

该 payload 应包含：

- `meta`
- `latest_quote`（适用时）
- `current_metrics`
- `trace_points`
- `recent_events`

之所以把 `trace_points` 放在 `recent_events` 前面，是因为用户明确要求先看连续数值，再看事件总结。

这样做的原因：

- 为所有输出格式提供同一份上游契约。
- 避免产品接口直接绑定在“DataFrame 列不断膨胀”的形态上。
- 方便未来扩展，而不必让每种输出格式都从增强后的表格里反推语义结构。

备选方案：

- 继续只扩展 DataFrame 列。放弃，因为无法建立稳定的多格式观察契约。
- 让渲染层直接从增强后的 DataFrame 推断结构。放弃，因为会让渲染层与指标细节和命令特定启发式强耦合。

### 决策二：在 enrichment 之后增加独立的事件检测层

实现应在指标增强之后、渲染之前，执行近期事件检测。

事件检测层应消费最近窗口内的归一化历史数据，输出结构化事件记录，至少包含：

- `bars_ago`
- `event_key`
- `subject_a`
- `relation`
- `subject_b`（适用时）
- 对比前值/现值（适用时）
- 英文事实描述

这样做的原因：

- enrichment 已经产出了所需指标序列。
- 事件检测和指标计算属于不同职责。
- 将事件逻辑独立出来，可以避免把 CLI 特定语义污染到纯计算指标函数中。

备选方案：

- 把事件标记直接编码进指标函数。放弃，因为指标函数应保持低耦合、可复用。
- 把事件推导写到 renderer 里。放弃，因为渲染层职责应是格式化，而不是推导市场现象。

### 决策三：支持多指标家族事件，但默认输出只陈列事实

实现应覆盖趋势、动量、波动、通道、量价/资金流等多类指标事件。但默认输出只陈列观测到的现象，不做方向性标签汇总。

这样做的原因：

- 用户需要的已经不只是交叉事件。
- 事实型事件记录更便于后续自行解释和程序消费。
- 不做预判性标签，可以降低歧义，也符合用户明确要求。

备选方案：

- 默认输出 bullish/bearish 计分卡。放弃，因为用户已经明确不要，而且此类标签会把多重假设压缩成不透明结论。

### 决策四：将“计算窗口”和“展示窗口”分离

实现应保留现有较长的历史窗口以保证指标稳定计算，同时新增独立的 trace window 作为展示窗口，默认值为 32。

这样做的原因：

- 很多指标计算所需历史长度大于用户想直接阅读的窗口。
- 指标稳定计算和近期展示是两个不同问题。

备选方案：

- 直接复用当前 `history_window` 作为展示窗口。放弃，因为用户明确要求默认展示 32 根，而现有 level 窗口是 60/120/200。

### 决策五：`trace_points` 允许横向数值块，其余信息优先纵向

结构化观察视图中的大多数 section 应采用纵向布局，但 `trace_points` 可以横向展开，以便在一个块内对比最近若干根的浮点数序列。

约束如下：

- `trace_points` 只能输出浮点数值，不使用字符画、sparkline 或视觉暗示符号
- `trace_points` 作为 section 放在 `recent_events` 前面
- `trace_points` 可按 block 切分，例如每 8 根一块

这样做的原因：

- 用户希望 traces 保留连续对比能力
- 纯纵向列出 32 根会过长，横向数值块更适合扫描趋势
- 仍然使用浮点数，可避免图形化误导

备选方案：

- 所有 section 一律纵向。放弃，因为会让 trace 阅读效率明显下降。
- 为 trace 使用字符图或 sparkline。放弃，因为用户明确不要。

### 决策六：table 输出使用统一的 boxed section layout

实现应为结构化观察视图新增一套独立的 box renderer，不再依赖 `DataFrame.to_string()` 作为主要 table 输出手段。

boxed section 的规则：

- 每个 section 都必须有完整、规整的 ASCII 矩形框
- section 标题位于外框顶部
- section 内部按条目列出内容
- `recent_events` 必须采用“一个总外框内列出多条事件”的形式，不能是一事件一外框
- `trace_points` 允许多个 boxed block，但每个 block 自身也必须是规整矩形

这样做的原因：

- 现有 pandas 风格宽表不适合 observation-oriented 输出
- 统一外框风格可以显著提升终端中的可扫读性和一致性
- `recent_events` 采用总外框，可以减少碎片化视觉噪音

备选方案：

- 继续使用宽 DataFrame 表格。放弃，因为在 trace 和事件并存时会不可读。
- 允许不同 section 使用不同外框风格。放弃，因为会降低一致性。

### 决策七：边框宽度必须基于折行后的内容动态计算

实现必须保证任何一行可见内容都不允许顶出右边框。为此，渲染流程应先做折行，再计算最终墙宽。

规则包括：

- 优先按空格折行
- 无空格时，按 `_`、`-`、`,`、`/`、`:` 等分隔符折行
- 如仍无法折行，则允许硬切分
- 折行后的续行应保持可读缩进
- 最终外框宽度应依据“折行后的最长可见行”动态计算，而不是原始最长字符串

这样做的原因：

- 用户明确要求任何字符串都不允许顶穿边框
- 如果直接按原始最长字符串定宽，超长 description 会把整块表格撑得过宽

备选方案：

- 仅按原始最长字符串计算外框宽度。放弃，因为会导致极宽输出。
- 允许内容裁剪。放弃，因为会丢失信息，不满足等价信息量要求。

### 决策八：多格式保持等价信息量，但接受不同序列化形态

实现应保证 `table`、`json`、`csv`、`tsv` 的信息量等价，但不要求长得一样。

具体做法：

- `table`：使用 boxed section layout
- `json`：使用结构化 schema
- `csv` / `tsv`：使用纵向归一化长表

这样做的原因：

- 不同格式天然适合不同形态
- 用户关心的是信息量一致，而不是视觉形态一致
- 长表 CSV/TSV 比把 traces 塞进单元格字符串更可用

备选方案：

- 强求四种格式完全同构。放弃，因为 CSV/TSV 很难无损承载嵌套结构且保持可用性。

### 决策九：新的结构化观察输出路径中，CLI 文案统一使用英文

新路径中的 section 名、字段名、event relation、event description、序列化 key 和导出列名都应统一使用英文。

这样做的原因：

- 用户已经明确要求 CLI 输出语言统一为英文
- 英文字段名对程序消费更稳定

备选方案：

- table 用中文、json/csv/tsv 用英文。放弃，因为会破坏统一契约。

## 风险 / 取舍

- [风险] 新的结构化 payload 会与现有 raw/enriched 路径形成双轨行为。 → 缓解：先限定到明确支持的命令，并为两条路径都补回归测试。
- [风险] “所有潜在多空信号”如果全部直接平铺，会产生噪音。 → 缓解：检测范围可以广，但默认 recent event list 只输出近期窗口内的事实记录。
- [风险] 纵向 section + boxed 布局更清晰，但整体输出会更长。 → 缓解：通过 section 划分和 trace block 分块提升可导航性。
- [风险] 动态宽度和不可穿墙约束会显著提高 table renderer 复杂度。 → 缓解：将 boxed renderer 设计成独立布局层，并通过专门回归测试覆盖折行与宽度计算。
- [风险] CSV/TSV 消费方可能习惯一行一个标的的扁平表。 → 缓解：将结构化观察模式下的 CSV/TSV 明确文档化为长表契约。
- [风险] CLI 英文输出与仓库其他本地化内容可能存在认知断层。 → 缓解：把英文约束明确限定在新的结构化观察输出路径。

## 迁移计划

1. 在不删除现有 raw/enriched 路径的前提下，引入结构化 observation payload builder。
2. 增加用于选择 observation 模式和配置 `trace_window` 的运行时选项。
3. 在 enrichment 之后实现近期事件检测与 trace 提取。
4. 为 `table`、`json`、`csv`、`tsv` 分别实现基于共享 payload 的渲染路径。
5. 为 payload 结构、事件检测、boxed table 渲染、动态宽度、折行行为、long-form CSV/TSV 导出补齐回归测试。
6. 在新路径稳定之前，保留旧路径作为回退机制。

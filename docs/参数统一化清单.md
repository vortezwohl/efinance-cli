# efinance-cli 参数统一化清单

## 1. 文档目标

本文档定义 `efinance-cli` 的目标参数体系。

本版本不考虑历史兼容，不讨论“旧参数是否保留”，只讨论：

- 参数应该如何命名
- 同类参数应该如何统一
- 哪些参数命名应当废弃
- 参数语义如何做到更直观、更好记、更少歧义

本文档的定位不是“迁移说明”，而是“目标规范”。

## 2. 设计原则

参数统一化必须遵守以下原则：

### 2.1 同一语义只允许一个主名

- 同一个概念不能出现两个参数名
- 不能让用户猜测“这两个是不是一回事”
- 不能依赖文档解释来弥补命名混乱

### 2.2 参数命名必须语义化

- 优先使用完整、可直接理解的英文单词
- 避免底层缩写直接暴露给 CLI 用户
- 避免过度依赖上游 Python API 的内部命名
- 参数名应当让用户不看源码也能猜到含义

### 2.3 参数命名必须可记忆

- 同类参数要共享同一命名风格
- 优先使用高频心智模型
- 尽量避免单词差一个字母的易混参数

### 2.4 运行时参数必须全局统一

- 输出控制参数必须统一
- 刷新控制参数必须统一
- 观察视图参数必须统一

### 2.5 业务参数要按语义分层

- 检索参数
- 时间参数
- 市场参数
- 数量参数
- 缓存/容错参数

每一层都必须采用稳定命名规则。

## 3. 统一后的目标参数体系

## 3.1 输出参数

以下参数定义为全局统一输出参数：

| 参数 | 语义 |
| --- | --- |
| `--format` | 输出格式 |
| `--full` | 完整输出，不截断宽列和长文本 |
| `--transpose` | 转置输出 |
| `--no-index` | 不输出索引列 |
| `--limit` | 仅输出前 N 行 |
| `--output` | 输出到文件 |
| `--encoding` | 文件编码 |

统一规则：

- 所有命令都应支持这一组参数，除非命令本身没有输出结果
- `--limit` 只表示“输出裁剪”，不表示业务查询数量
- `--output` 只表示写文件，不承担导出格式选择职责

## 3.2 刷新参数

以下参数定义为全局统一刷新参数：

| 参数 | 语义 |
| --- | --- |
| `--watch` | 开启刷新模式 |
| `--interval` | 刷新间隔秒数 |
| `--count` | 刷新次数 |
| `--clear` / `--no-clear` | 刷新前是否清屏 |

统一规则：

- `--count` 只能表示刷新次数
- 任何业务参数都不得再使用 `count`
- 任何“查询数量”“返回数量”“下载数量”都必须使用专用语义名

## 3.3 观察视图参数

以下参数定义为全局统一观察视图参数：

| 参数 | 语义 |
| --- | --- |
| `--view` | 输出视图模式 |
| `--indicator-level` | 指标增强等级 |
| `--trace-window` | observation 追踪窗口大小 |

统一规则：

- 所有支持 observation 的命令都必须支持这三项
- 顶层入口与底层入口不得出现能力面不一致

## 4. 按语义分组后的目标命名

## 4.1 市场参数

### 目标主名

- `--market`

### 说明

CLI 面向用户时，`market` 比 `market-type` 更短、更自然、更好记。

`type` 在这里没有提供额外价值，反而增加噪音。

### 规范

- 所有市场筛选类参数统一使用 `--market`
- 不再使用 `--market-type`
- `--market` 的取值是市场枚举名，而不是自由文本说明

### 示例

- `--market us_stock`
- `--market hongkong`
- `--market a_stock`

## 4.2 时间参数

### 目标主名

- `--date`
- `--start-date`
- `--end-date`
- `--dates`

### 规范

- 单一日期使用 `--date`
- 日期范围使用 `--start-date` 和 `--end-date`
- 多个离散日期使用 `--dates`

### 禁止项

- 不再使用 `--beg`
- 不再使用 `--end`

### 原因

- `beg` 不直观
- `start-date` / `end-date` 更符合自然语言
- `date` / `dates` 虽然相近，但语义边界清晰，且比 `beg/end` 更容易记忆

## 4.3 检索参数

### 目标主名

- `--query`
- `--result-count`

### 规范

- 面向搜索类能力时，关键词主名统一为 `--query`
- 返回候选数量统一为 `--result-count`

### 适用范围

- 搜索证券
- 搜索 quote id
- 模糊检索类命令

### 说明

当前有的位置使用 `keyword`，有的位置把搜索值放在位置参数里。

为了提高一致性，理想状态下搜索命令都应统一为：

- `search --query AAPL`
- `utils search-quote --query AAPL --result-count 10`

如果保留位置参数，也应保证内部语义仍然是 `query`。

## 4.4 标的参数

### 目标主名

- `--symbol`
- `--symbols`
- `--quote-id`
- `--quote-ids`

### 规范

- 面向股票、基金、债券、期货等“用户可识别代码”时，统一使用 `symbol`
- 单个标的：`--symbol`
- 多个标的：`--symbols`
- 只有在底层明确要求东方财富行情 ID 时，才使用 `--quote-id`
- 多个行情 ID：`--quote-ids`

### 禁止项

- 不再按市场类型拆成：
  - `stock_code`
  - `fund_code`
  - `bond_code`
  - `index_code`
  - `codes`
  - `stock_codes`
  - `fund_codes`
  - `bond_codes`

### 原因

- 这些命名把“领域类型”重复写进参数名，增加记忆负担
- 命令路径本身已经表达了领域
- 在 `fund ...` 命令里再写 `fund_code` 是冗余的

### 示例

- `stock get-base-info --symbol 600519`
- `fund get-base-info --symbol 161725`
- `common get-latest-quote --quote-id 105.AAPL`

## 4.5 数量参数

数量参数必须严格按语义拆分。

### 目标主名

| 参数 | 语义 |
| --- | --- |
| `--count` | 刷新次数 |
| `--result-count` | 搜索候选数量 |
| `--limit` | 输出显示行数 |
| `--top` | 业务排名前 N 项 |
| `--max-records` | 明细或数据抓取上限 |
| `--max-files` | 文件下载上限 |

### 禁止项

- 不再使用泛化的 `--max-count`

### 原因

- `max-count` 在不同命令里含义不一致
- 有时是明细条数
- 有时是下载文件数
- 同名异义会降低可读性

### 规范

- 如果语义是“记录条数上限”，使用 `--max-records`
- 如果语义是“下载文件数量上限”，使用 `--max-files`
- 如果语义是“排行前 N 个”，使用 `--top`
- 如果语义是“仅展示前 N 行”，使用 `--limit`

## 4.6 缓存参数

### 目标主名

- `--use-cache`
- `--use-local-cache`
- `--use-id-cache`

### 规范

- 所有缓存策略参数统一使用肯定式
- 不使用否定式主名

### 禁止项

- 不再使用 `--no-cache`

### 原因

- 否定式更难理解默认行为
- 多个否定参数叠加时认知成本更高
- `use-*` 风格更统一

### 进一步规则

- 若只是泛指是否使用缓存，用 `--use-cache`
- 若明确是本地检索缓存，用 `--use-local-cache`
- 若明确是 quote id 缓存，用 `--use-id-cache`

## 4.7 容错参数

### 目标主名

- `--ignore-errors`

### 禁止项

- 不再使用 `--suppress-error`

### 原因

- `ignore-errors` 更符合用户心智
- `suppress-error` 更像内部实现措辞
- “ignore” 比 “suppress” 更自然

### 规范

- 若命令允许跳过局部失败并继续处理，统一使用 `--ignore-errors`
- 若命令不支持该策略，就不暴露该参数

## 4.8 路径参数

### 目标主名

- `--output`
- `--output-dir`

### 规范

- 输出单文件路径使用 `--output`
- 输出目录使用 `--output-dir`

### 禁止项

- 不再使用 `--save-dir`

### 原因

- `output-dir` 与 `output` 命名成对，更统一
- `save-dir` 不如 `output-dir` 清晰

## 5. 顶层入口与底层入口统一规则

顶层入口与底层入口不能再各自定义一套几乎相同但略有差异的参数面。

### 目标规则

- 顶层 `search` 与 `utils search-quote` 应共享同一组核心语义参数
- 若二者都属于搜索能力，则至少必须统一：
  - `--query`
  - `--market`
  - `--result-count`
  - 全套输出参数
  - 全套刷新参数
  - 全套观察视图参数

### 不允许的状态

- 顶层命令叫 `--market`，底层命令叫 `--market-type`
- 顶层命令不支持 `--indicator-level`，底层支持
- 顶层命令不支持 `--view`，底层支持

## 7. 应废弃的参数命名

以下参数命名在目标规范中应视为废弃：

- `--market-type`
- `--beg`
- `--end`
- `--no-cache`
- `--suppress-error`
- `--save-dir`
- `--max-count`
- `stock_code`
- `fund_code`
- `bond_code`
- `index_code`
- `codes`
- `stock_codes`
- `fund_codes`
- `bond_codes`
- `quote_id_list`

## 8. 优先级清单

### 第一优先级

- 统一市场参数为 `--market`
- 统一时间参数为 `--date / --start-date / --end-date / --dates`
- 统一标的参数为 `--symbol / --symbols / --quote-id / --quote-ids`
- 统一数量参数分工，去掉泛化 `--max-count`
- 补齐顶层入口和底层入口的共有参数面

### 第二优先级

- 统一缓存参数为 `--use-*`
- 统一容错参数为 `--ignore-errors`
- 统一目录输出参数为 `--output-dir`

### 第三优先级

- 全量梳理帮助文案
- 全量梳理 README 和 `docs/` 示例
- 为所有命令生成正式参数字典

## 9. 验收标准

统一化完成后，必须满足以下标准：

- 同一语义只有一个主名
- CLI 中不再出现明显底层缩写主名
- 同类命令的参数风格一致
- 顶层命令与底层命令不再各说各话
- 数量类参数不会再出现同名异义
- 市场、日期、缓存、容错参数都形成稳定命名体系
- 用户无需反复查文档即可猜出大多数参数名

## 10. 一句话总结

最终目标不是“把上游 API 原样搬到 CLI”，而是把 CLI 参数体系收敛成一套：

- 好记
- 好猜
- 好读
- 少歧义
- 可全局复用

的稳定命名规范。

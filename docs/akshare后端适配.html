Multi-Backend Architecture Design

efinance-cli 多后端能力架构设计：从单一 efinance CLI 到后端无关金融命令平台

本文档用于指导 efinance-cli 从“围绕 efinance 上游函数组织命令”的单后端 CLI， 演进为“以能力、命令语义和结果契约为中心”的多后端金融命令平台。本文档不再以旧版本兼容为首要目标， 而是直接采用更干净的命令模型、后端模型与执行模型，为当前的 efinance、akshare， 以及未来可能接入的 yfinance 预留稳定扩展空间。

当前结论 应重建命令模型，而不是继续围绕 efinance 函数做兼容壳
核心变化点 后端会增加、能力不对齐、字段不对齐、专属命令长期存在
目标状态 共享语义层后端无关，扩展命令层允许 provider-specific 差异
设计参考 设计模式文章
旧版本兼容不是目标 共享命令层必须后端无关 不同后端能力不完全一致是正常前提 未来预留 yfinance 扩展空间

1. Design Contract

设计定约

Objective

  • 建立后端无关的共享命令层。
  • 允许 efinance、akshare、未来 yfinance 通过统一能力协议接入。
  • 允许后端声明额外能力、额外命令与额外字段，而不是强行压平差异。
  • 让 observation、enrichment、rendering 依赖结果契约，而不是依赖具体 provider。

Non-goals

  • 不追求所有后端输出完全同构。
  • 不追求一开始覆盖当前全部命令与全部品类。
  • 不追求首版做成插件市场或运行时热插拔系统。
  • 不把 provider-specific 语义伪装成共享语义。

Constraints

  • 共享命令必须稳定面向“能力语义”而不是“第三方函数名”。
  • 命令参数模型不能继续绑定上游函数签名。
  • 能力缺失必须显式暴露，而不是静默降级成错误语义。
  • 所有新结构必须可测试、可枚举、可落到 contract test。
收束判断： 当前项目最稳妥的目标不是“给现有 efinance CLI 增加 akshare 支持”，而是“把 CLI 的稳定对象从上游函数切换为能力、命令语义和结果契约”。

2. Current Diagnosis

当前架构为什么不适合继续原地扩展

当前仓库已经有命令树、执行器、渲染、observation 与 enrichment 分层，但关键绑定点仍然是 efinance 函数。问题不只在“调用的是哪个后端”，还在“命令长什么样、参数怎么长出来、增强怎么回补” 都依赖同一个上游函数对象。

当前层 现状 为什么会卡住多后端
命令注册 以 ALLOWED_FUNCTIONS + CLI_PATH_OVERRIDES 管理上游函数白名单 稳定对象仍是 efinance.xxx.yyy，不是能力语义
参数生成 Click 参数由 callback 签名反射生成 CLI 参数模型直接绑定上游函数签名，无法让共享命令独立演化
执行调用 执行器直接调用 request.spec.callback(**kwargs) Facade 层尚未形成，调用目标仍是第三方函数对象
enrichment 历史回补直接调用 efinance.*.get_quote_history 即使主命令切后端，增强链也可能悄悄回退到 efinance
observation 依赖模块名、原始字段别名和历史回补函数 未真正退化为“只消费标准结果契约”
关键结论： 现状不适合继续沿着“旧命令 + --backend + 两个适配器”的方式渐进扩展。因为最大的耦合不只在调用层，还在参数层和结果增强层。

3. Architectural Position

新版架构立场

共享语义必须后端无关

例如 equity price history、fund nav history、instrument search 这类命令，应首先表达业务语义，而不是表达来源库。

能力差异必须显式存在

不是每个后端都能返回相同数据，也不是每个后端都支持同一能力。共享层不能伪装不存在的能力。

扩展命令必须合法共存

akshare macro、未来 yfinance option 这类 provider-specific 能力应进入扩展命令树，而不是污染共享命令树。

最终目标形态： efinance-cli 不再只是“自然化的 efinance CLI”，而是“以金融数据能力为中心的多后端 CLI 平台”。efinance、akshare、yfinance 都只是 provider。

4. Patterns

设计模式在本项目中的精确用法

模式 落点 用途
Adapter 每个 provider 的 capability handler 把 efinance / akshare / yfinance 的真实 API 翻译成统一请求与结果契约
Strategy capability route 同一共享命令在运行时按 backend 选择不同 handler，而不是在命令内部写条件分支
Facade command facade CLI 只面对一个高层入口，不直接接触 provider 与标准化细节
Factory Method backend provider / handler 创建 调用方不负责实例化具体适配器
Proxy 能力检查、限流、缓存、重试、fallback 把访问控制逻辑放在调用边界，而不是散落在命令处理里
Decorator schema validation、raw capture、tracing、metrics 在不改 handler 接口的前提下叠加横切行为
Template Method 统一执行骨架 固定“校验 -> 路由 -> 调用 -> 标准化 -> enrich -> observe -> render”链路

5. Target Structure

目标结构草图

下面的草图不是实现细节，而是项目新的稳定骨架。

CLI / TUI
  -> Command Catalog
  -> Request Schema / Validation
  -> Command Facade
  -> Capability Router
  -> Backend Provider
  -> Capability Handler Adapter
  -> Result Contract / Standardization
  -> Enrichment / Observation / Rendering

共享稳定对象:
  - capability_id
  - request schema
  - result contract
  - execution pipeline

不再作为稳定对象的内容:
  - efinance.stock.get_quote_history
  - akshare.stock_zh_a_hist
  - yfinance.Ticker.history
设计边界： 共享命令层只关心“我要哪种能力”，provider 层才关心“具体调哪个第三方函数”。这是本次重构最核心的边界切换。

6. Command Model

命令模型草图：共享命令树与扩展命令树并存

共享命令树

equity price history
equity price snapshot
equity price live
equity profile
fund nav history
fund profile
instrument search

共享命令树面向稳定业务语义。命令本身不暴露 provider 名称，但允许通过 --backend 选择具体来源。

provider-specific 扩展命令树

akshare macro cpi
akshare industry board
akshare option chain
yfinance option chain
yfinance earnings calendar

扩展命令树面向后端独有能力，不伪装成共享命令，也不强迫其他后端实现。

关键原则： 共享层负责“同类能力的稳定入口”，扩展层负责“释放 provider 独有价值”。两者都走同一执行骨架，但不共享错误语义。

7. Object Sketch

核心对象草图

下面这组对象是建议的最小核心模型。重点不是字段名本身，而是职责边界。

from dataclasses import dataclass, field
from typing import Any, Protocol


@dataclass(slots=True)
class CommandDefinition:
    command_id: str
    cli_path: tuple[str, ...]
    capability_id: str | None
    request_schema: str
    help_text: str
    backend_scope: str  # shared | provider_specific
    allowed_backends: set[str] = field(default_factory=set)
    provider_name: str | None = None
    supports_watch: bool = False


@dataclass(slots=True)
class CapabilityDescriptor:
    capability_id: str
    domain: str
    result_contract: str
    required_fields: tuple[str, ...]
    optional_fields: tuple[str, ...] = ()


@dataclass(slots=True)
class BackendContext:
    backend_name: str
    raw_mode: bool = False
    timeout_seconds: float | None = None


@dataclass(slots=True)
class CommandRequest:
    command_id: str
    capability_id: str | None
    business_args: dict[str, Any]
    output_args: dict[str, Any]
    backend: BackendContext


class CapabilityHandler(Protocol):
    capability_id: str

    def invoke(self, request: CommandRequest) -> Any:
        ...


class BackendProvider(Protocol):
    name: str

    def supports(self, capability_id: str) -> bool:
        ...

    def get_handler(self, capability_id: str) -> CapabilityHandler:
        ...
这组对象解决的问题： 命令不再依赖上游函数签名；能力不再要求所有后端实现同一个胖接口；provider-specific 命令也可以在统一 request / render 骨架内合法存在。

8. Result Contract

结果契约草图：不是一张万能大表，而是一组 capability contract

当前 observation 与 enrichment 之所以难以解耦，是因为它们同时要理解模块名、原始字段别名和上游回补逻辑。 新方案中，不应追求“一个标准字段宇宙统一所有结果”，而应按能力定义契约。

HistoryBarContract

  • 必需字段：symbol、timestamp、open、high、low、close
  • 可选字段：volume、turnover、change_pct
  • 适用能力：历史 K 线、部分 snapshot 回补

RealtimeQuoteContract

  • 必需字段：symbol、name、last_price
  • 可选字段：open、high、low、volume
  • 适用能力：实时列表、最新快照

ProfileContract / SearchContract

  • 必需字段更少，但允许原始字段通过 extensions 保留
  • 避免为统一而过度裁剪后端独有信息
重要边界： 结果标准化的目标不是保留所有原始字段，而是给共享后处理链提供稳定核心字段；provider 扩展字段可以通过 extensions 或 raw 视图保留。

9. Execution Flow

执行链路草图

1. CLI 解析命令路径
2. Command Catalog 找到 CommandDefinition
3. request schema 校验并构建 CommandRequest
4. Capability Router 解析 backend
5. backend.supports(capability_id) 做能力检查
6. backend.get_handler(capability_id) 获取适配器
7. handler.invoke(request) 调用上游
8. result contract standardizer 标准化结果
9. enrichment 只消费 contract，不再直接 import provider
10. observation 只消费 contract / enriched result
11. rendering 输出 raw / observation / csv / tsv / json
12. watch 模式重复执行同一骨架
直接收益： 统一执行链会继续保留，但调用目标从“第三方函数”切换为“能力处理器”，这让双后端、多后端、provider-specific 命令都能进同一骨架。

10. Backend Strategy

面向未来 yfinance 的预留策略

未来接入 yfinance 时，最容易犯的错误是提前设计一个过胖的 MarketBackend 超大接口， 然后让每个 provider 填满一长串 NotImplemented。这会让接口失真，也会把 provider 差异压成噪音。

方案 短期感受 长期效果
胖接口后端对象 看起来统一 随着 akshare / yfinance 差异增大，会充满无意义空实现
capability handler registry 初期多一点样板 可以只为真实支持的能力注册 handler，更适合多后端长期演化
建议： 后端模型采用 BackendProvider + CapabilityHandler，而不是“一个实现全部方法的大对象”。这正是为未来 yfinance 预留可持续扩展空间。

11. Migration

实施顺序建议

由于不再把旧版本兼容作为首要约束，实施顺序可以更果断，优先切断最深的耦合点。

  1. 先重建 CommandDefinition、CommandRequest 与 request schema，不再让 CLI 参数从上游函数签名反射生成。
  2. 再建立 CapabilityDescriptor、BackendProvider 与 CapabilityHandler 骨架。
  3. 首批只落一组共享能力，例如 stock.price.history、stock.profile、instrument.search。
  4. 先接 efinance provider，证明新骨架可跑。
  5. 再接 akshare 同能力 handler，并补 contract test。
  6. 再改造 enrichment / observation，让它们从“看 provider”切换为“看 contract”。
  7. 最后增加 provider-specific 扩展命令树。
不建议的顺序： 不建议先在现有命令树上硬加 --backend，再去回头重写参数模型与能力模型。那样会让旧耦合延续更久，返工更大。

12. Verification

验证策略草图

Schema Test

验证每个共享命令的 request schema 是否能独立于 provider 存在，参数错误是否在路由前被拦截。

Contract Test

对同一个 capability，验证不同 backend 的结果都满足对应 contract 的必需字段、类型和排序约束。

Capability Matrix Test

验证支持矩阵、能力缺失报错、provider-specific 命令隔离与帮助页提示。

Pipeline Regression

验证 raw / observation / csv / tsv / json、watch、enrichment 与 rendering 在双后端下继续走同一执行骨架。

13. Final Position

最终建议

一句话总结： 本项目适合演进为“语义稳定、后端可替换、能力分层清晰、允许 provider-specific 扩展”的多后端金融 CLI，不适合继续围绕 efinance 函数做兼容式生长。
  • 第一原则：共享命令层后端无关，provider 差异显式存在。
  • 第二原则：命令、参数、结果都围绕 capability contract 组织，而不是围绕第三方函数组织。
  • 第三原则：共享能力与扩展能力分树而治，但统一进入同一执行骨架。
  • 第四原则：未来接入 yfinance 时，不扩胖接口，而是增加 handler 与 contract。

本文档为当前架构设计稿，重点在于重新定义稳定边界、能力模型与迁移方向，不把“未实现”表述为“已完成且可直接上线”。

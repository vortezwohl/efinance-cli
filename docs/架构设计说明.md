# 架构设计说明

## 1. 文档定位

本文档说明当前 `efinance-cli` 的运行时架构。当前系统已经完成从“第三方函数驱动 CLI”到“能力驱动多后端 CLI”的切换，运行时主链只围绕以下稳定对象组织：

- `CommandDefinition`
- `RequestSchema`
- `CapabilityDescriptor`
- `BackendProvider`
- `CapabilityHandler`
- `ResultContract`

用户命令、请求校验、结果标准化、增强、observation 与渲染，都应围绕这些稳定对象扩展，而不是重新引入对第三方函数名的直接绑定。

## 2. 当前架构目标

当前架构主要解决四类变化点：

1. 后端会持续增加  
   当前已接入 `efinance`、`akshare` 与 `yfinance`。
2. 不同后端支持的能力集合不一致  
   共享能力与 provider 扩展能力需要长期共存。
3. 同一能力的参数与返回结构不一致  
   需要显式的 request schema 与结果契约做收敛。
4. 渲染、增强与 observation 不能继续绑死某个 provider  
   需要统一执行骨架作为收敛点。

## 3. 架构分层

当前实现可以概括为以下分层：

```text
CLI
-> Command Catalog
-> Request Schema
-> Backend Resolver
-> Command Facade
-> Backend Provider
-> Capability Handler
-> Result Contract / Standardization
-> Enrichment / Observation
-> Rendering
```

### 3.1 命令目录层

文件：

- `efinance_cli/command_catalog.py`
- `efinance_cli/commands.py`

职责：

- 定义共享命令的稳定对象；
- 定义命令键、CLI 路径、支持矩阵和帮助元数据；
- 以显式 request schema 生成 Click 参数；
- 挂载共享命令与 provider 扩展命令，不再挂载旧函数驱动命令树。

当前已经落地的共享命令包括：

- `instrument.search`
- `stock.price.history`
- `stock.price.live`
- `stock.profile`
- `fund.nav.history`

### 3.2 后端层

文件：

- `efinance_cli/backends/base.py`
- `efinance_cli/backends/factory.py`
- `efinance_cli/backends/resolver.py`
- `efinance_cli/backends/providers.py`

职责：

- 注册 `BackendProvider`；
- 解析 `--backend` 与默认 backend 语义；
- 暴露共享 capability handler；
- 暴露 provider 扩展命令定义；
- 对可选依赖缺失、限流与不支持场景暴露显式错误。

当前约束如下：

- 共享命令默认 backend 为 `auto`，候选链固定为 `akshare -> yfinance -> efinance`；
- provider 扩展命令默认 backend 为所属 provider；
- backend 选择冲突会显式失败；
- `yfinance` 已正式注册为 backend provider；
- `yfinance` 共享能力当前覆盖搜索、历史、最新价 / 快照、股票 / 基金 / quote 资料；
- `yfinance` 通过 `quote news` 暴露首批 Yahoo 专属扩展命令；
- provider 扩展命令在未指定 backend 与显式传入 `--backend auto` 时，都会自动适配到所属 provider，而不是进入完整 failover 链；
- auto 成功命中后会把 `final_backend` 继续暴露给 enrichment、observation 与 raw 输出元信息；
- `YFRateLimitError` 与依赖缺失会被显式转换为用户可读错误。

### 3.3 执行与门面层

文件：

- `efinance_cli/facade.py`
- `efinance_cli/executor.py`

职责：

- 接收统一请求对象；
- 执行 schema 校验、backend 解析和 capability 路由；
- 调用对应 handler；
- 将标准化结果继续送入增强、observation 与渲染。

当前 `CommandExecutor` 只服务于 shared / provider-extension 命令路径，不再承载旧函数驱动执行分支。

### 3.4 结果契约与标准化层

文件：

- `efinance_cli/contracts.py`
- `efinance_cli/backends/providers.py`

职责：

- 定义共享能力的核心字段；
- 定义 provider 原始字段到标准字段的映射；
- 区分核心字段、可选字段、provider 字段和 metadata；
- 对缺字段、非法 payload 与不支持场景抛出显式标准化错误。

当前已覆盖并被 `yfinance` 复用的结果契约包括：

- `search-results`
- `history-bars`
- `profile-info`
- `fund-nav-history`
- `realtime-quotes`
- `provider-records`

### 3.5 Enrichment / Observation 层

文件：

- `efinance_cli/enrichment/service.py`
- `efinance_cli/observation.py`

职责：

- 基于稳定 command key、结果契约与标准补充接口进行增强和结构化观察；
- 对共享命令通过标准补充接口回补历史数据；
- 对 provider-extension 与未纳入专用模板的结果统一走 generic observation 兜底；
- 保留 raw 输出中的 provider 原始字段，便于排查兼容差异。

当前这一层已经不再依赖旧的 `(module_name, function_name)` 分类常量，也不再保留直连旧 provider 回补路径的内部入口。

## 4. 命令分类策略

当前命令分为两类：

### 4.1 共享命令

共享命令面向后端无关的稳定语义。它们必须具备：

- 显式 request schema；
- 显式支持矩阵；
- 标准结果契约；
- 统一执行骨架接入。

当前架构中，shared 还有一条额外硬约束：只有 `supported_backends` 数量大于等于 2 的命令才允许进入 shared catalog。只支持单一 backend 的命令，即使继续沿用稳定业务路径暴露，也必须转入 provider-extension。

### 4.2 Provider 扩展命令

provider 扩展命令用于表达特定后端的专属能力。它们允许：

- 只属于某个 provider；
- 挂载到稳定的业务语义命令树；
- 复用统一执行骨架；
- 使用轻量扩展结果契约，而不强行塞进共享最小公分母。

当前示例命令包括：

- `bond catalog`
- `quote price latest`
- `search local`
- `stock industry boards`

## 5. 当前采用的设计模式

当前实现并不追求“模式越多越好”，而是只在真实变化点使用最小必要模式。

### 5.1 Facade

`CommandFacade` 负责把命令请求收敛到 provider 与 capability 调用。

### 5.2 Strategy

不同 backend 的 capability handler 按运行时 backend 选择执行。

### 5.3 Factory Method

`backends/factory.py` 负责集中创建并暴露 provider。

### 5.4 Adapter

各 provider handler 把 `efinance`、`akshare` 的真实 API 结果适配到统一结果契约。

### 5.5 Template Method

统一执行骨架固定为：

1. 构建请求
2. 校验 schema
3. 解析 backend
4. 调用 handler
5. 标准化结果
6. 执行 enrichment
7. 生成 observation
8. 渲染输出

## 6. BREAKING 变化

当前架构相对于旧设计的核心 breaking change 如下：

1. 命令定义不再围绕第三方函数签名生成。
2. 共享命令参数不承诺与某个 provider 的原始函数参数一一对应。
3. 帮助页、测试和运行时校验都以 capability 与 request schema 为中心。
4. provider 专属能力不再强行混入共享命令树。
5. 后端不支持时会显式失败，而不是静默回退。

## 7. 当前实现状态

截至当前版本，已完成：

- 多后端共享命令目录骨架；
- 各 provider 基础注册与解析；
- 首批共享结果契约；
- `search`、`stock history/live/profile`、`fund nav history` 的多后端适配；
- `quote history/latest/profile`、`fund profile` 的 `yfinance` 适配；
- `quote news` 的 `yfinance` provider-extension 适配；
- `stock` / `fund` / `bond` / `futures` 与 `quote` / `market` / `resolve` 的完整命令目录迁移，以及单 backend 命令向 provider-extension 的重分类；
- provider 扩展命令示例；
- provider 层负例与约束测试；
- 运行时 legacy 阴影与无效 import 的第一轮系统性清理。

补充说明：

- `quote news` 使用 `--result-count` 作为业务参数，避免与统一 watch 运行时 `--count` 产生冲突；
- `yfinance` 的真实网络验证应视为可选 smoke，而不是稳定测试基线，因为 Yahoo 限流会直接影响结果可重复性。

当前未覆盖的内容不是 legacy 命令主链，而是未来能力扩展，例如：

- 新共享 capability 的继续迁移；
- 更多 provider-extension 命令；
- `yfinance` 的期权链、基金画像细分视图与更多 Yahoo 专属能力。

## 8. 完整支持与支持矩阵约束

当前架构里，“命令存在”与“完整支持”是两件不同的事。只有同时满足下面四层闭环，某个 capability 才能被声明为 supported：

1. 命令定义、请求 schema 与 backend 解析都能稳定进入执行骨架；
2. provider handler 返回稳定的标准结果契约，而不是宽松原始对象；
3. enrichment / observation / rendering 会把该命令送入明确的主链或受约束的 generic 兜底；
4. 测试中的支持矩阵与 provider 真实实现一致。

本轮实现后，关键主链已经扩展到：

- 多资产历史序列：`stock.price.history`、`bond.price.history`、`futures.price.history`、`quote.price.history`、`fund.nav.history`、`fund.nav.history-batch`
- 多资产资料回补：`stock.profile`、`fund.profile`、`bond.profile`、`quote.profile`
- 多资产实时回补：`stock.price.live`、`bond.price.live`、`futures.price.live`、`quote.price.latest`

`flow` / `trades` / `catalog` 等记录类命令虽然暂不进入专用 observation 模板，但也必须经过显式契约化和 generic observation 兜底，不能再停留在“只要能路由就算支持”的状态。

## 9. 维护约束

如果后续继续扩展能力，应按以下顺序判断：

1. 这是共享能力还是 provider 专属能力？
2. 是否需要新增稳定命令键与 request schema？
3. 是否需要新增或扩展结果契约？
4. 是否可以复用现有 enrichment / observation？
5. 是否可以通过标准补充接口完成二次取数，而不是重新引入旧式 provider 回补分支？

不要再新增“直接绑定第三方函数名”的用户命令，也不要重新引入旧函数驱动命令树。

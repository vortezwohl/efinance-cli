# 架构设计说明

## 1. 目标

本项目需要在 `D:\github-project\efinance-cli\efinance_cli` 下实现一套优雅、可扩展、
面向 Agent 的命令行结构。

这里的“优雅”不是指层数越多越好，而是指：

- 变化点清晰分层；
- 依赖方向单一；
- 能快速追加新的命令暴露；
- 运行时表格输出、刷新与导出逻辑不污染业务命令；
- 后续 Agent 阅读代码时可以快速定位职责。

## 2. 采用的结构思路

结合 AI Coding 时代对设计模式“按需使用”的原则，当前实现只采用最少但有价值的模式化结构。

### 2.1 Facade：注册中心 `registry.py`

`efinance` 是第三方包，若命令层直接在各处散着访问：

- 容易出现不同文件对同一个模块做不一致处理；
- 后续新增帮助文本、过滤可暴露函数时会分散修改；
- 模块名、函数名与点击命令之间的映射不集中。

因此使用一个轻量 Facade：

- 集中定义允许暴露的模块；
- 集中定义允许暴露的函数；
- 集中维护帮助文本和副作用标记；
- 对外提供 `build_command_specs`。

### 2.2 Strategy：渲染层 `rendering.py`

`efinance` 返回值类型并不统一，常见有：

- `DataFrame`
- `Series`
- `dict[str, DataFrame]`
- `list`
- `namedtuple`
- `None`

如果把所有分支堆在命令回调里，后续维护会非常差。

因此渲染层以“按类型选择渲染策略”的方式集中处理：

- `render_dataframe`
- `render_mapping`
- `render_sequence`
- `render_json`
- `render_csv`

这样后续如果需要接入 Rich 表格或更复杂的格式，只需改渲染层。

### 2.3 模板化执行骨架：`executor.py`

所有命令执行都遵循同一条骨架：

1. 接收原始 CLI 参数
2. 拆分业务参数与运行时参数
3. 按函数签名做类型转换
4. 调用第三方函数
5. 渲染结果
6. 输出到控制台或写入文件
7. 如启用 `watch`，则循环重复

这是一种模板化执行思路，不需要刻意写成复杂继承体系，也能保持流程统一。

## 3. 目录与职责

### 3.1 `main.py`

负责对外暴露程序主入口，不承载业务逻辑。

### 3.2 `app.py`

负责装配根命令，连接入口和命令构建层。

### 3.3 `commands.py`

负责：

- 构建根命令组
- 构建模块命令组
- 动态生成函数命令
- 给所有命令挂统一运行时选项

### 3.4 `registry.py`

负责：

- 暴露模块名
- 暴露函数白名单
- 生成 `CommandSpec`
- 标记副作用与是否允许 `watch`

### 3.5 `introspection.py`

负责从 Python 函数签名生成 CLI 参数，并在执行前做轻量类型转换。

### 3.6 `executor.py`

负责执行管线和循环刷新。

### 3.7 `rendering.py`

负责所有结果格式的输出渲染。

### 3.8 `models.py`

负责共享数据模型，避免跨模块传递松散字典。

## 4. 为什么不直接把所有逻辑写进一个 Click 文件

如果只看“当前代码量”，一个大文件当然也能跑。

但这会立刻带来几个问题：

- 命令暴露逻辑和渲染逻辑耦合；
- 后续新增输出格式需要修改命令定义文件；
- `watch` 逻辑容易复制到多个命令；
- 未来要增加语义别名、错误处理、缓存策略时扩展点不清晰。

当前分层虽然比单文件多，但每层职责明确，规模仍然可控，没有过度设计。

## 5. 扩展方式

### 5.1 新增上游函数暴露

只需改 `registry.py` 中对应模块的白名单。

### 5.2 新增帮助文本

修改 `FUNCTION_HELP_OVERRIDES`。

### 5.3 新增输出格式

在 `rendering.py` 中增加新渲染函数，并在 `render_value` 中接入。

### 5.4 新增参数转换规则

在 `introspection.py` 中扩展 `coerce_parameter_value`。

### 5.5 新增语义化快捷命令

可以在 `commands.py` 中额外添加顶层别名命令，而不需要破坏现有模块命令树。

## 6. 当前风险

### 6.1 第三方包数据源不稳定

当前 CLI 直接封装 `efinance`，而 `efinance` 自身依赖外部网络数据源。
因此本项目最主要的不稳定性并不来自 CLI 架构，而来自外部接口。

### 6.2 上游文档与源码存在乱码

当前环境中的 `efinance 0.5.8` 源码和元数据存在中文乱码，因此：

- 不能完全信任上游 docstring；
- 帮助文本需要逐步人工覆盖；
- 中文文档应由本项目自己维护。

### 6.3 自动签名映射仍然是“宽松适配”

为了优先覆盖全部公开 API，当前参数反射采用了保守、宽松的规则。
这保证了覆盖面，但并不意味着所有复杂参数组合都已经被精细校验。

## 7. 后续建议

如果继续演进，建议优先做以下事项：

1. 为高频命令补充专项参数帮助与示例
2. 对常见网络错误做统一错误包装
3. 为表格输出增加更稳定的宽度与分页控制
4. 为 `search`、`quote-id`、`quote-history` 组合场景补充语义化快捷命令
5. 增加最小自动化测试，覆盖命令树生成与渲染层行为

当前结构已经足够支撑这些后续演进，不需要推翻重来。

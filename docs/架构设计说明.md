# 架构设计说明

## 1. 文档定位

本文档说明当前 `efinance-cli` 的多后端命令架构，而不是旧版本的函数驱动包装架构。

当前系统的稳定抽象对象已经从：

- `efinance.xxx.yyy` 上游函数；

迁移为：

- `CommandDefinition`
- `RequestSchema`
- `CapabilityDescriptor`
- `BackendProvider`
- `CapabilityHandler`
- `ResultContract`

CLI 的目标也随之变化为：围绕金融数据能力组织命令，而不是围绕某个第三方库的函数树组织命令。

## 2. 当前架构目标

当前架构主要解决四类变化点：

1. 后端会持续增加  
   当前已接入 `efinance` 与 `akshare`，并为 `yfinance` 预留挂载点。

2. 不同后端支持的能力集合不一致  
   共享能力与 provider 扩展能力必须同时存在。

3. 同一能力的参数与返回结构不一致  
   需要显式 request schema 与结果契约做标准化。

4. 现有渲染、增强、observation 不能继续直接绑死某个 provider  
   需要统一执行骨架作为收敛点。

## 3. 架构分层

当前实现可以概括为以下分层：

```text
CLI
-> Command Catalog / Legacy Registry
-> Request Schema
-> Backend Resolver
-> Command Facade
-> Backend Provider
-> Capability Handler
-> Result Contract / Standardization
-> Enrichment / Observation
-> Rendering
```

各层职责如下。

### 3.1 命令目录层

文件：

- `efinance_cli/command_catalog.py`
- `efinance_cli/commands.py`

职责：

- 定义共享命令的稳定对象；
- 定义命令键、CLI 路径、支持矩阵和帮助元数据；
- 以显式 request schema 生成 Click 参数；
- 在迁移期同时挂载共享命令、provider 扩展命令和旧函数驱动命令。

当前已经落地的共享命令包括：

- `instrument.search`
- `equity.price.history`
- `equity.price.live`
- `equity.profile`
- `fund.nav.history`

### 3.2 后端层

文件：

- `efinance_cli/backends/base.py`
- `efinance_cli/backends/factory.py`
- `efinance_cli/backends/resolver.py`
- `efinance_cli/backends/providers.py`

职责：

- 注册 `BackendProvider`；
- 解析 `--backend` 与默认 backend 语义；
- 暴露共享 capability handler；
- 暴露 provider 扩展命令定义；
- 为未来 optional provider 预留挂载点。

当前约束如下：

- 共享命令默认 backend 为 `efinance`；
- provider 扩展命令默认 backend 为所属 provider；
- backend 选择冲突会显式失败；
- `yfinance` 当前只在 `list_optional_provider_names()` 中作为预留 provider 暴露。

### 3.3 执行与门面层

文件：

- `efinance_cli/facade.py`
- `efinance_cli/executor.py`

职责：

- 接收统一请求对象；
- 执行 schema 校验、backend 解析和 capability 路由；
- 调用对应 handler；
- 将标准化结果继续送入增强、observation 与渲染。

当前 `CommandExecutor` 同时承载：

- 旧函数驱动路径；
- 新的 shared/provider-extension 路径。

这是迁移期的必要折中，不是最终稳态。

### 3.4 结果契约与标准化层

文件：

- `efinance_cli/contracts.py`
- `efinance_cli/backends/providers.py`

职责：

- 定义共享能力的核心字段；
- 定义 provider 原始字段到标准字段的映射；
- 区分核心字段、可选字段、provider 字段和 metadata；
- 对缺字段、非法 payload 与不支持场景抛出显式标准化错误。

当前已覆盖的结果契约包括：

- `search-results`
- `history-bars`
- `profile-info`
- `fund-nav-history`
- `realtime-quotes`
- `provider-records`

### 3.5 Enrichment / Observation 层

文件：

- `efinance_cli/enrichment/service.py`
- `efinance_cli/observation.py`

职责：

- 基于标准结果契约进行增强和结构化观察；
- 逐步摆脱直接依赖 `efinance.*.get_quote_history` 的旧回补路径；
- 在迁移期同时兼容标准结果和部分旧中文列结果。

当前已经完成的关键收敛点：

- shared history 支持通过标准补充接口做回补；
- shared history / profile / realtime-list 可以进入统一 observation 链；
- raw 输出会保留 provider 原始字段，便于排查兼容差异。

## 4. 命令分类策略

当前命令分为三类：

### 4.1 共享命令

共享命令面向后端无关的稳定语义。它们必须具备：

- 显式 request schema；
- 显式支持矩阵；
- 标准结果契约；
- 统一执行骨架接入。

### 4.2 provider 扩展命令

provider 扩展命令用于表达特定后端的专属能力。它们允许：

- 只属于某个 provider；
- 使用独立命令根组；
- 复用统一执行骨架；
- 使用轻量扩展结果契约，而不强行塞进共享最小公分母。

当前示例命令是：

- `akshare industry boards`

### 4.3 旧函数驱动命令

旧函数驱动命令仍然存在，是因为：

- 当前还有大量命令尚未迁移；
- 回归测试与用户使用路径仍依赖它们；
- 迁移需要分 capability 逐步完成。

但这些命令不再是未来设计的中心对象，后续任务 `8.5` 会继续清理它们的专属兼容路径。

## 5. 当前采用的设计模式

当前实现并没有追求“模式越多越好”，而是只在真实变化点使用最小必要模式。

### 5.1 Facade

`CommandFacade` 负责把命令请求收敛到 provider 与 capability 调用。

### 5.2 Strategy

不同 backend 的 capability handler 按运行时 backend 选择执行。

### 5.3 Factory Method

`backends/factory.py` 负责集中创建并暴露 provider。

### 5.4 Adapter

各 provider handler 把 `efinance`、`akshare` 的真实 API 结果适配到统一结果契约。

### 5.5 Template Method

统一执行骨架固定为：

1. 构建请求
2. 校验 schema
3. 解析 backend
4. 调用 handler
5. 标准化结果
6. 执行 enrichment
7. 生成 observation
8. 渲染输出

## 6. BREAKING 变化

当前架构相对于旧设计的核心 breaking change 如下：

1. 命令定义不再围绕第三方函数签名生成。
2. 共享命令参数不承诺与某个 provider 的原始函数参数一一对应。
3. 帮助页、测试和运行时校验都以 capability 与 request schema 为中心。
4. provider 专属能力不再强行混入共享命令树。
5. 后端不支持时会显式失败，而不是静默回退。

## 7. 当前迁移状态

截至当前版本，已完成：

- 多后端共享命令目录骨架；
- 双 provider 基础注册与解析；
- 首批共享结果契约；
- shared history / profile / realtime / search / fund nav 的双后端适配；
- provider 扩展命令示例；
- provider 层负例与约束测试。

仍未完成：

- CLI 回归测试整体重写；
- 旧函数驱动路径的系统性清理；
- 某些 observation 字段兼容逻辑的进一步下沉。

## 8. 维护约束

如果后续继续扩展能力，应按以下顺序判断：

1. 这是共享能力还是 provider 专属能力；
2. 是否需要新增稳定命令键与 request schema；
3. 是否需要新增或扩展结果契约；
4. 是否可以复用现有 enrichment / observation；
5. 是否必须触碰旧函数驱动兼容路径。

不要再新增“直接绑定第三方函数名”的用户命令。

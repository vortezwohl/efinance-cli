主线判断 不是零散加命令，而是从函数驱动 CLI 迁到多后端命令运行时

1. Thesis

一句话结论

这 15 个提交做的不是“在原有 CLI 上顺手加个 akshare”，而是一次完整的命令运行时迁移： 从“直接反射 efinance 函数签名生成 Click 命令”，迁移为“命令目录 + request schema + capability + result contract + backend/provider” 驱动的多后端 CLI。

因此最重要的区别不在命令数量，而在命令定义权发生了转移。 基线版本里，命令的稳定对象还是 efinance.stock.get_quote_history 这类上游函数； 现在稳定对象已经变成 stock.price.history、fund.nav.history、 instrument.search 这类命令键和能力语义。

2. Inventory

先看最终总差异

命令路径层面

  • 基线版本有 50 条业务命令路径。
  • 当前版本有 52 条叶子路径，其中 1 条是 watch，1 条是新增业务命令 stock industry boards。
  • 旧的 50 条业务路径最终全部保留，没有在最终 HEAD 中永久消失。

帮助页层面

  • 旧版帮助页围绕“对应函数: efinance.module.function”。
  • 新版 shared 命令帮助页围绕“命令键 / 能力标识 / 支持后端 / 命令类别”。
  • 也就是说，帮助页已经不再把第三方函数视为一等公民。

参数与执行层面

  • 旧版大部分参数由 Python 函数签名动态反射。
  • 新版 shared 命令参数来自显式 RequestSchema。
  • 新版引入统一 --backend，并开始对 backend 约束做显式校验。
最容易误判的一点： 如果只看最终命令路径，会以为“几乎没怎么改”；但如果看帮助页、参数定义来源、错误提示、raw/observation 输出契约，就会发现这其实已经不是同一套 CLI 运行时了。

5. Exhaustive Deltas

最终差异穷举

命令面

  • 最终保留基线全部 50 条业务路径。
  • 最终净新增 1 条业务命令：stock industry boards。
  • 中间态曾新增 instrument search、equity price history、equity profile、equity price live、akshare industry boards，但最终已被回摆或重挂载。

帮助与参数

  • shared 命令帮助页新增命令键、能力标识、支持后端。
  • shared 命令参数不再依赖第三方函数签名。
  • 缺参报错从“函数签名侧漏”变成“schema 驱动的可读错误”。

运行时

  • 新增 backends/ provider 注册与 resolver。
  • 新增 result contract 和字段别名归一化。
  • enrichment / observation 逐步从 legacy 回补转向标准历史补充接口。
如果必须用最准确的话描述这段历史： “命令面尽量维持用户熟悉的业务语义，但命令的来源、执行、结果标准化和观察链已经彻底从单后端思维迁到了多后端 runtime 思维。”

6. Special Focus

特别说明：为什么看起来没怎么加命令，却其实变化极大

因为这次迁移的目标从来不是“让用户多记几条新命令”，而是把命令定义权从第三方函数手里拿回来。 一旦定义权转移完成，后续新增 provider、接入扩展命令、统一结果契约、修 observation 字段兼容，就都不需要再围着 efinance 某个函数的签名打补丁。

对使用者而言，这种迁移最危险的地方恰恰是“路径没怎么变，所以容易低估风险”。 实际上帮助页、参数、输出字段、backend 行为、错误提示的细节都已经发生了结构性变化。 
旧世界：
命令 = 上游函数 + CLI_PATH_OVERRIDES + introspection

新世界：
命令 = command key + request schema + capability + backend selection + result contract

7. Final Assessment

最后的工程判断

从工程质量看，这段历史最难得的地方不是“代码量很大”，而是它没有停在最危险的中间态。 项目一度确实进入过 instrument / equity / akshare 这种过渡命令面，但最终又把表层收束回 stock / fund / bond / futures / quote / market / resolve 这套用户更容易理解的业务命令树。

换句话说，这不是一次单纯的“内部换血”，也不是一次只追求表面稳定的“保守兼容”。 它做的是更难的那件事：在底层 runtime 大迁移之后，把最终用户看到的命令语言重新打磨回业务语义。

最终判断： 相对 0c100203，当前版本最本质的变化不是“多了哪些命令”，而是“命令已经不再从第三方函数自然长出来”， 而是从仓库自己维护的命令目录、能力契约和后端运行时长出来。
